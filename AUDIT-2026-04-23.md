## Audit 1 — Labeler / scorer / dictionary consistency (H3 analog)

### Tools with overlapping judgment scope

| Tool | Input | Judgment mechanism | Dictionary? |
|------|-------|---------------------|-------------|
| `measurement/score_auto.py` | 3 shuffled outputs + signature | **Claude-as-judge** (Opus 4.7) | NO keyword dict — delegates to LLM prompt |
| `measurement/score.py` | 3 shuffled outputs | **Human subjective pick** | NO dict — picks stored as A/B/C |
| `skills/extract/extract.py` | samples corpus | **Claude-as-extractor** (Opus 4.7) with structured-JSON schema output | NO keyword dict — delegates to LLM + schema |
| `skills/review/review.py` | new signature vs permanent signature | **Deterministic set-diff** with normalized `item_key()` | N/A — no classification dictionary, just identity keys |
| `agents/signature-brutus.md` | signature + samples (F1) or arbitrary user work (F2) | **Claude-as-reviewer** (Opus, dispatched) | NO dict — prompt-driven |
| `agents/signature-qa.md` | signature + schema (F1) or arbitrary code (F2) | **Claude-as-QA** (Haiku, dispatched) | NO dict |
| `agents/signature-historian.md` | signature + history (F1) or arbitrary timeseries (F2) | **Claude-as-classifier** (Sonnet, dispatched) | NO dict |
| `skills/advisor/advisor.py` | context at inflection | **Claude-as-advisor** (Opus) | NO dict — prompt-driven |

### Key finding

**No keyword dictionaries exist anywhere in the CogSig pipeline.** Every judgment tool delegates scoring/classification to an LLM call (either inline Claude via SDK, or dispatched through Managed Agents). The one place identity comparison happens — `review.py`'s `item_key()` function — is a single implementation consumed by all review commands, so there's no drift risk from multiple normalizers.

The H3 failure class in private-scaffold was *two different Python keyword dicts* in `pattern_qvalue.py` vs `pattern_reliability.py` producing sign-disagreement. That architecture does not exist here — there's no multi-tool Python-regex keyword lookup in CogSig.

### Single shared identity key in review.py

```python
# skills/review/review.py:67-89
def _normalize(text, limit=80):
    norm = " ".join(text.lower().split())
    return norm[:limit].strip(" ,.;:!?")

def item_key(dim, item):
    if dim == "recurring_decision_templates":
        return f"{dim}::{_normalize(item.get('situation',''))}||{_normalize(item.get('response',''))}"
    if dim == "recurring_failure_patterns":
        return f"{dim}::{_normalize(item.get('pattern',''), 100)}"
    # ...
```

All five callers within review.py (diff_signature, cmd_approve, cmd_reject, cmd_edit, and load_rejected's implicit match) use the same `item_key`. Consistent.

### Judge-prompt content (score_auto.py)

The Claude-as-judge prompt template is stable + explicit + not keyword-based:
- Reads the signature JSON (what traits to look for)
- Reads 3 shuffled outputs
- Asks for A/B/C pick with confidence + reasoning
- Strict JSON output

Re-running 10/10 identically today (Audit 3b result) demonstrates judge stability under the fixed seed + deterministic shuffle + same-model-identifier.

### Sign-convention risk

The only multi-rater scenario is signature-brutus + signature-qa + signature-historian reviewing the same signature. These produce ORTHOGONAL outputs (adversarial-verdict vs schema-validity vs drift-classification), not overlapping judgments on the same event. There is no "same event, different sign" risk because they don't score the same dimension.

### Verdict

**LABELER-AUDIT-STATUS: CONSISTENT**

CogSig's architecture avoids the H3 failure class by design. Every judgment tool either:
1. Delegates to an LLM (no local dictionary to drift from another tool's dictionary), or
2. Uses deterministic set-diff via a single shared `item_key` implementation.

The only architectural risk analogous to H3 would be if someone later adds a second normalizer or a parallel Python-regex-based classifier. Not present today.

### MUST-DO before next public release

**Fix-1 (normie auto-promote)**: In `review.py::cmd_refresh_queue`, after the first-run seed branch, add a steady-state branch: `if preset == "normie" and pending: auto_apply_to_permanent(pending); pending = []`. About 10 lines. Preserves the "silent hands-off" promise.

This also solves Fix-3.

**Fix-3 (managed-agents auto-trigger)**: Call `managed-agents/review.py` as subprocess from `extract.py` when `active_mode == "cloud"`. Or stop claiming "automatically triggered" in SUBMISSION.md line 70 and reframe as "available via `python3 managed-agents/review.py`."

**Fix-4 (governance-gates-permanent-signature)**: The README claim that Brutus+QA+Historian gate permanent signature entry needs either code wiring OR a prose rewrite. Wiring means approve() dispatches to each agent and refuses if any returns FAIL. Prose rewrite drops the claim and keeps agents available-on-demand.

### Priorities

Fix-1 is the most user-facing gap — a normie user's permanent signature won't grow past first-extract. Fix-4 is the most submission-critical — it's a claim about architecture that doesn't match the code. Fix-2 + Fix-3 travel together: either wire the mode selector or drop the triadic mode framing.

### What CogSig got RIGHT that private-scaffold got WRONG

Several architectural decisions inoculate CogSig against the cited-not-invoked failure class:

1. **LLM-delegated judgment everywhere** — no two Python tools compete on overlapping keyword dicts, so H3-style labeler-drift is impossible by construction.
2. **Single shared `item_key()` normalizer** — identity keys are computed one way, consumed everywhere. In the private scaffold, two different `_normalize_text()` functions in pattern_qvalue.py vs pattern_reliability.py caused sign disagreement at n=40% items.
3. **Advisor-at-inflection IS actually wired** — the advisor mechanism is in the class of "declared and implemented," unlike the private scaffold's cited-not-invoked agent-breadcrumb-emit rule. Extract.py line 285+ reaches the advisor module and writes reports when triggered.
4. **Managed Agents registration is real** — the three governance agents exist server-side (direct-verified by retrieve()) with full system prompts. This is architectural infrastructure that WAS built, even if the auto-trigger step from extract → review.py is the cited-not-invoked gap.
5. **Empirical validation is real** — the 10/10 auto-scorer result reproduces exactly, meaning the pipeline between signature extraction → judge-call → accuracy measurement has no drift.

The failure class in CogSig is concentrated at the **operator-surface layer** (presets, modes, auto-triggers described in user-facing docs) — NOT at the **pipeline-core layer** (extract → review-queue → inject). Core works. Peripherals overclaim.

### Fix-4 — Governance gate on `approve()` (LANDED)

`skills/review/review.py::cmd_approve` now runs `_qa_validate_patterns()` BEFORE writing to permanent — refuses with exit=1 if any pattern fails required-field / type / instance_count>=2 / non-empty evidence_list checks. The QA gate is deterministic stdlib-only ($0, fast).

Validated: malformed pending item (evidence_list removed) → approve refused with precise error. Valid pending item → approve succeeds.

README line 52 reframed from "every pattern passes Brutus+QA+Historian" to accurate per-mode phrasing: "every pattern passes QA schema validation; in cloud mode, Brutus+Historian also review via Managed Agents; in team mode, in-session subagents are invokable at the prompt."
